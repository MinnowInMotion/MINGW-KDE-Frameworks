31 ## Introduction
32 
33 *Internationalization* (i18n) is the process of preparing
34 the source code such that the running program can receive and
35 output user-visible information in various human languages, or,
36 more precisely, *locales*. This is because the same language may
37 be spoken at several places which differ in other details of relevance
38 to the software internationalization (e.g. dialect, calendar, etc).
39 I18n is performed by the programmer.
40 *Localization* (l10n) is the process of adapting relevant
41 program resources for a particular locale.
42 L10n is performed by the translator, who is usually a native of the locale.
43 
44 User interface text is the most prominent resource needing l10n.
45 In the source code, there are two general approaches
46 to writing i18n'd user-visible strings, hereafter called *messages*.
47 In the first approach, the programmer writes symbolic identifier in place
48 of actual message text, and then for each language there is a file,
49 called translation *catalog*, which contains the actual text linked
50 to the appropriate identifier.
51 For example, in the code:
52 
53 ~~~
54 QString msg = getId("out_of_disk_space");
55 ~~~
56 
57 and in the translation catalog:
58 
59 ~~~
60 out_of_disk_space "There is no space left on the disk."
61 ~~~
62 
63 In the second approach, the programmer writes actual text inside
64 the source code, in an agreed upon human language,
65 which is usually American English. Then there is a translation catalog
66 for every *other* language, in which English and translated text of
67 the message are linked. For example, in the code:
68 
69 ~~~
70 QString msg = i18n("There is no space left on the disk.");
71 ~~~
72 
73 and in the translation catalog:
74 
75 ~~~
76 msgid "There is no space left on the disk."
77 msgstr "Nema više prostora na disku."
78 ~~~
79 
80 One may observe that the second approach is technically a generalization
81 of the first approach. However, in actual use there are non-negligible
82 differences in the i18n/l10n workflow between these approaches,
83 and therefore they are treated as qualitatively different.
84 
85 Both approaches -- the symbolic-identifier and the actual-text -- have
86 some advantages and disadvantages over each other. That is why today
87 both are in use, in software i18n in general. Within free software,
88 though, the actual-text approach is considerably more widespread,
89 as embodied in the GNU Gettext i18n system. Whether this is a historical
90 accident, or due to sensibilities of people writing free software,
91 is left to the interested reader to research.
92 
93 The KDE Ki18n library, being built atop Gettext, provides
94 the actual-text approach. In the following it is described
95 how to use Ki18n, from the viewpoint of programmers and maintainers.
96 Basic instructions are split into three parts:
97 
98 \li The first part deals with how to write i18n messages in the code.
99  This is the largest and the most important part, since there are
100  many nuances to preparing quality messages for translation.
101  Section: \ref write_i18n.
102 
103 \li The second part describes how to connect particular translation calls
104  with particular translation catalogs, so that the system can know
105  in which catalogs to search for translations.
106  In particular, there are some differences here depending on whether
107  the piece of code is an application or a library.
108  Section: \ref link_cat.
109 
110 \li The third part explains how to *extract* messages from the code,
111  in order to create the translation *template*.
112  The template is simply a clean translation catalog (having only English
113  part of each message), which the translator copies over and fills in
114  the translations for a particular language.
115  Then it is shown how and where to install translation catalogs,
116  and how to update existing translation catalogs to newly extracted
117  templates (after some new development has been done, causing
118  new messages to appear, some to be removed, and some modified).
119  Unlike in the GNU Gettext manual, here there are no dependencies or
120  references to a particular software build system;
121  this is left to the maintainer to choose.
122  Section: \ref handle_cat.
123 
124 There are also two special topics:
125 
126 \li Some programmers like to have more structure and consistency
127  in message texts, and for them Ki18n provides
128  a customizable semantic markup.
129  Section: \ref kuit_markup.
130 
131 \li Sometimes there are program resources other than text
132  that require localization.
133  Ki18n provides a generic, though rudimentary solution for such cases.
134  Section: \ref non_text.

139 ## Writing Messages
140 
141 Most messages can be internationalized with simple `i18n*` calls,
142 which are described in the section \ref gen_usage.
143 A few messages may require treatment with `ki18n*` calls,
144 and when this is needed is described in the section \ref spec_usage.
145 Argument substitution in messages is performed using the familiar
146 Qt syntax `%<number>`, but there may be some differences;
147 the section \ref subs_notes gives some notes about placeholders.
148 Finally, guidelines for writing good messages are given
149 in sections \ref good_text and \ref good_ctxt.
150 
151 <a name="gen_usage">
152 
153 ### General Messages
154 
155 General messages are wrapped with `i18n*` calls.
156 These calls are *immediate*, which means that they return
157 the final localized text (including substituted arguments)
158 as a `QString` object, that can be passed to UI widgets.
159 
160 The most frequent message type, a simple text without any arguments,
161 is handled like this:
162 
163 ~~~
164 QString msg = i18n("Just plain info.");
165 ~~~
166 
167 The message text may contain arbitrary Unicode characters,
168 and the source file *must* be UTF-8 encoded.
169 Ki18n supports no other character encoding.
170 
171 If there are some arguments to be substituted into the message,
172 `%<number>` placeholders are put into the text at desired positions,
173 and arguments are listed after the string:
174 
175 ~~~
176 QString msg = i18n("%1 has scored %2", playerName, score);
177 ~~~
178 
179 Arguments can be of any type for which there exists an overloaded
180 `KLocalizedString::subs` method.
181 Up to 9 arguments can be inserted in this fashion, due to the fact that
182 `i18n` calls are realized as overloaded templates.
183 If more than 9 arguments are needed, which is extremely rare,
184 a `ki18n*` call (described later) must be used.
185 
186 Sometimes a short message in English is ambiguous to translators,
187 possibly leading to a wrong translations.
188 Ambiguity can be resolved by providing a context string along the text,
189 using the `i18nc` call. In it, the first argument is the context,
190 which only the translator will see, and the second argument is the text
191 which the user will see:
192 
193 ~~~
194 QString msg = i18nc("player name - score", "%1 - %2", playerName, score);
195 ~~~
196 
197 Section \ref good_ctxt gives a few pointers on when contexts are needed,
198 and what they should contain.
199 
200 In messages stating how many of some kind of objects there are,
201 where the number of objects is inserted at run time, it is necessary
202 to differentiate between *plural forms* of the text.
203 In English there are only two forms, one for number 1 (singular) and
204 another form for any other number (plural).
205 In other languages this might be more complicated (more than two forms),
206 or it might be simpler (same form for all numbers).
207 This is handled properly by using the `i18np` plural call:
208 
209 ~~~
210 QString msg = i18np("%1 image in album %2", "%1 images in album %2",
211  numImages, albumName);
212 ~~~
213 
214 The plural form is decided by the first integer-valued argument,
215 which is `numImages` in this example. In rare cases when there are
216 two or more integer arguments, they should be ordered carefully.
217 
218 In QML code, due to some limitations, always the first argument
219 will be treated as plural-deciding, so it should be an appropriate
220 number (integer or a round double); otherwise, behavior is undefined.
221 
222 It is also allowed to omit the plural-deciding placeholder, for example:
223 
224 ~~~
225 QString msg = i18np("One image in album %2", "%1 images in album %2",
226  numImages, albumName);
227 ~~~
228 
229 or even:
230 
231 ~~~
232 QString msg = i18np("One image in album %2", "More images in album %2",
233  numImages, albumName);
234 ~~~
235 
236 If the code context is such that the number is always greater than 1,
237 the plural call must be used nevertheless.
238 This is because in some languages there are different plural forms
239 for different classes of numbers; in particular, the singular form
240 may be used for numbers other than 1 (e.g. those ending in 1).
241 
242 If a message needs both context and plural forms, this is provided by
243 `i18ncp` call:
244 
245 ~~~
246 QString msg = i18ncp("file on a person", "1 file", "%1 files", numFiles);
247 ~~~
248 
249 
250 In the basic `i18n` call (no context, no plural) it is not allowed
251 to put a literal string as the first argument for substitution.
252 In debug mode this will even trigger a static assertion,
253 resulting in compilation error. This serves to prevent misnamed calls:
254 context or plural frequently needs to be added at a later point to
255 a basic call, and at that moment the programmer may forget to update
256 the call name from `i18n` to `i18nc/p`.
257 
258 Furthermore, an empty string should never be wrapped with
259 a basic `i18n` call (no `i18n("")`),
260 because in translation catalog the message with empty text
261 has a special meaning, and is not intended for client use.
262 The behavior of `i18n("")` is undefined,
263 and there will be some warnings in debug mode.
264 
265 There is also a complement of `i18nd*` calls
266 (`i18nd`, `i18ndc`, `i18ndp`, `i18ndcp`),
267 which are not supposed to be used directly,
268 but as will be explained in the section \ref link_cat.

271 
272 ### Specialized Messages
273 
274 There are some situations where `i18n*` calls are not sufficient,
275 or are not convenient enough.
276 One obvious case is if more than 9 arguments need to be substituted.
277 Another case is if it would be easier to substitute arguments later on,
278 after the line with the i18n call.
279 For cases such as these, `ki18n*` calls can be used.
280 These calls are *deferred*, which means that they do not return
281 the final translated text as `QString`, but instead return
282 a `KLocalizedString` instance which needs further treatment.
283 Arguments are then substituted one by one using
284 `KLocalizedString::subs` methods, and after all arguments
285 have been substituted, the translation is finalized by one of
286 `KLocalizedString::toString` methods (which return `QString`).
287 For example:
288 
289 ~~~
290 KLocalizedString ks;
291 case (reportSource) {
292  SRC_ENG: ks = ki18n("Engineering reports: %1"); break;
293  SRC_HEL: ks = ki18n("Helm reports: %1"); break;
294  SRC_SON: ks = ki18n("Sonar reports: %1"); break;
295  default: ks = ki18n("General report: %1");
296 }
297 QString msg = ks.subs(reportText).toString();
298 ~~~
299 
300 
301 `subs` methods do not update the `KLocalizedString` instance on which
302 they are invoked, but return a copy of it with one argument slot filled.
303 This allows to use `KLocalizedString` instances as a templates
304 for constructing final texts, by supplying different arguments.
305 
306 Another use for deferred calls is when special formatting of arguments
307 is needed, like requesting the field width or number of decimals.
308 `subs` methods can take these formatting parameters.
309 In particular, arguments should not be formatted in a custom way,
310 because `subs` methods will also take care of proper localization
311 (e.g. use either dot or comma as decimal separator in numbers, etc):
312 
313 ~~~
314 // BAD (number not localized):
315 QString msg = i18n("Rounds: %1", myNumberFormat(n, 8));
316 // Good:
317 QString msg = ki18n("Rounds: %1").subs(n, 8).toString();
318 ~~~
319 
320 
321 Like with `i18n`, there are context, plural, and context-plural
322 variants of `ki18n:`
323 
324 ~~~
325 ki18nc("No function", "None").toString();
326 ki18np("File found", "%1 files found").subs(n).toString();
327 ki18ncp("Personal file", "One file", "%1 files").subs(n).toString();
328 ~~~
329 
330 
331 `toString` methods can be used to override the global locale.
332 To override only the language of the locale, `toString` can take
333 a list of languages for which to look up translations
334 (ordered by decreasing priority):
335 
336 ~~~
337 QStringList myLanguages;
338 ...
339 QString msg = ki18n("Welcome").toString(myLanguages);
340 ~~~
341 
342 The section \ref link_cat describes how to specify
343 the translation *domain*, a canonical name for the catalog file
344 from which `*i18n*` calls will draw translations.
345 But `toString` can always be used to override the domain for a given call,
346 by supplying a specific domain:
347 
348 ~~~
349 QString trName = ki18n("Georgia").toString("country-names");
350 ~~~
351 
352 Relevant here is the set of `ki18nd*` calls
353 (`ki18nd`, `ki18ndc`, `ki18ndp`, `ki18ndcp`),
354 which can be used for the same purpose,
355 but which are not intended to be used directly.
356 This will also be covered in the section \ref link_cat.

360 #### Dynamic Contexts
361 
362 Translators are provided with the capability to script translations,
363 such that the text changes based on arguments supplied at run time.
364 For the most part, this feature is transparent to the programmer.
365 However, sometimes the programmer may help in this by providing
366 a *dynamic* context to the message, through
367 `KLocalizedString::inContext` methods.
368 Unlike the static context, the dynamic context changes at run time;
369 translators have the means to fetch it and use it to script
370 the translation properly. An example:
371 
372 ~~~
373 KLocalizedString ks = ki18nc("%1 is user name; may have "
374  "dynamic context gender=[male,female]",
375  "%1 went offline");
376 if (knownUsers.contains(user) && !knownUsers[user].gender.isEmpty()) {
377  ks = ks.inContext("gender", knownUsers[user].gender);
378 }
379 QString msg = ks.subs(user).toString();
380 ~~~
381 
382 Any number of dynamic contexts, with different keys, can be added like this.
383 Normally every message with a dynamic context should also have
384 a static context, like in the previous example, informing the translator
385 of the available dynamic context keys and possible values.
386 Like `subs` methods, `inContext` does not modify the parent instance,
387 but returns a copy of it.

390 
391 ### Placeholder Substitution
392 
393 Hopefully, most of the time `%<number>` placeholders are substituted
394 in the way one would intuitively expect them to be.
395 Nevertheless, some details about substitution are as follows.
396 
397 Placeholders are substituted in one pass, so there is no need
398 to worry about what will happen if one of the substituted arguments
399 contains a placeholder, and another argument is substituted after it.
400 
401 All same-numbered placeholders are substituted with the same argument.
402 
403 Placeholders directly index arguments: they should be numbered from 1
404 upwards, without gaps in the sequence, until each argument is indexed.
405 Otherwise, error marks will be inserted into message at run time
406 (when the code is compiled in debug mode), and any invalid placeholder
407 will be left unsubstituted.
408 The exception is the plural-deciding argument in plural calls,
409 where it is allowed to drop its placeholder,
410 in either the singular or the plural text.
411 
412 If none of the arguments supplied to a plural call is integer-valued,
413  an error mark will be inserted into the message at run time
414  (when compiled in debug mode).
415 
416 Integer arguments will be by default formatted as if they denote
417 an amount, according to locale rules (thousands separation, etc.)
418 But sometimes an integer is a numerical identifier (e.g. port number),
419 and then it should be manually converted into `QString` beforehand
420 to avoid treatment as amount:
421 
422 ~~~
423 i18n("Listening on port %1.", QString::number(port));
424 ~~~

429 ### Writing Good Texts
430 
431 When writing message texts, sometimes it is tempting to assemble text
432 from pieces such as to have less repetition.
433 However, such shortcuts frequently cannot work for other languages,
434 and are almost always confusing to translators.
435 The first rule of writing good message texts is therefore to
436 *keep sentences together and clearly structured*
437 (or "no word puzzles"), even at the cost of some repetition.
438 
439 At its basic, this rule means always to use placeholders for insertion
440 of run time arguments, rather than string concatenation. For example:
441 
442 ~~~
443 // BAD:
444 i18n("File ") + filePath + i18n(" not found.");
445 // Good:
446 i18n("File %1 not found.", filePath);
447 ~~~
448 
449 This is rather obvious, since it also results in clearer and shorter code.
450 But sometimes placeholders can be overused:
451 
452 ~~~
453 // BAD:
454 i18n("This contact is now %1.",
455  isOnline ? i18n("online") : i18n("offline"));
456 // Good:
457 isOnline ? i18n("This contact is now online.")
458  : i18n("This contact is now offline.");
459 ~~~
460 
461 The shorter version here is bad, because the sentence structure of
462 translation may need to change by more than the one word,
463 and also because a less thorough translator may fail to check
464 in which way the short messages "online" and "offline" are used.
465 
466 If an otherwise long text needs to differ in only small part,
467 then a reasonable solution is to split it at sentence level,
468 but also explain the situation to translators through context:
469 
470 ~~~
471 // BAD:
472 i18n("Something very long here. This contact is now %1.",
473  isOnline ? i18n("online") : i18n("offline"));
474 // Good:
475 i18nc("%1 one of the 'This contact...' messages below",
476  "Something very long here. %1",
477  isOnline ? i18n("This contact is now online.")
478  : i18n("This contact is now offline."));
479 // BAD:
480  i18n("Something very long here. ")
481 + isOnline ? i18n("This contact is now online.")
482  : i18n("This contact is now offline.");
483 ~~~
484 
485 The third version above is bad because, firstly, the translator
486 may wonder about the trailing space in the first message or
487 simply overlook it, and secondly, there may be some cross-dependency
488 between the translation of the long message and the short messages.
489 In general, insertions of one message into another should *always*
490 be accompanied by contexts, and composition-significant
491 leading and trailing whitespace should be avoided.
492 
493 The second basic rule of writing good texts is to
494 *expose every user-visible text for translation*.
495 One should *never* make assumptions of the type
496 "this does not need translation" or "it will be same in every language".
497 
498 One example where programmers sometimes make such assumption
499 are compositions without any letters:
500 
501 ~~~
502 // BAD:
503 QString("%1: %2").arg(albumName, playCount);
504 // Good:
505 i18nc("album name: play count", "%1: %2", albumName, playCount)
506 ~~~
507 
508 Here, in some languages the arrangement of whitespace will need to differ
509 (e.g. a space may be needed before the colon as well).
510 Letter-free compositions should also normally be equipped with context,
511 because, for example, separation may depend on type of arguments.
512 
513 Another example of user-visible texts sometimes wrongly omitted
514 from i18n are proper names:
515 
516 ~~~
517 // BAD:
518 label1->setText(i18n("Written by:"));
519 label2->setText("Roofus McBane");
520 // Good:
521 label1->setText(i18n("Written by:"));
522 label2->setText(i18n("Roofus McBane"));
523 ~~~
524 
525 Proper names too may need localization, for example, transliteration
526 when the target language uses a different writing system.
527 This holds for proper names of people, and of anything else.
528 
529 When it comes to text markup, like the HTML subset supported by
530 Qt rich text processing, opinions are divided on how much of it
531 to expose for translation. One argument goes that markup may be
532 confusing for translators, and that exposing it increases
533 the chance of syntactical errors in translation.
534 This is true as such, but it should be balanced with situations where,
535 either, the translator needs to modify the markup,
536 or, the markup will convey some context to translator.
537 For example, typographical modifiers should always be left in:
538 
539 ~~~
540 // BAD:
541 label->setText("<b>" + i18n("Disk almost full:") + "</b>");
542 // Good:
543 label->setText(i18n("<b>Disk almost full:</b>"));
544 ~~~
545 
546 because the target language may have different typographical standards
547 (e.g. CJK languages tend to avoid boldface in text body font sizes,
548 as it makes ideographs harder to recognize).
549 Especially if tags are found around internal parts of the message text,
550 it would be ungainly to hide them, e.g. by placeholder insertion.
551 But, values to tag attributes, such as links in <tt><a href='...'></tt>,
552 tags, should be inserted through placeholders
553 (unless it is expected that translators provide localized values).
554 In this example:
555 
556 ~~~
557 // Good:
558 i18n("<p>Preprocessing has failed.</p>"
559  "<p>Please check your bracketed images stack.</p>"
560  "<p>Click \"Details\" to show processing details.</p>")
561 ~~~
562 
563 `<p>` tags could be avoided by splitting this message into
564 one message per sentence, but this should not be done, because
565 paragraphs should be translated as single units of meaning.
566 
567 Another important point about XML-like text markup, is to try and keep it
568 well-formed in XML sense on the level of standalone message. For example:
569 
570 ~~~
571 // BAD (for translation, although fine by HTML / Qt rich text):
572 i18n("<p>Some sentence."
573  "<p>Another sentence.");
574 // Good:
575 i18n("<p>Some sentence.</p>"
576  "<p>Another sentence.</p>");
577 ~~~
578 
579 Well-formedness is good because the most frequent error in translation
580 in presence of markup is mistyping (or miscopying) a tag.
581 If the original text is well-formed, a translation checker tool
582 can require the same of translation, and signal when that is not so.
583 The previous example of non-well-formedness was perhaps trivial;
584 in practice, non-trivial examples usually break some other rules too
585 (e.g. no word puzzles).

588 
589 ### Writing Good Contexts
590 
591 The message context, given as first argument in `*i18nc` calls,
592 is of great help to translators. Unfortunately, to a programmer it is
593 not always clear when a context is needed, or what it should state.
594 So, the very first rule of writing good contexts is to
595 *listen to the translators asking for contexts*.
596 When taking suggestions from translators, there is no need to worry if
597 the proposed context will be sufficient for "all" languages.
598 It is fine to simply add the information that a translator into
599 particular language requested, and wait for translators into other
600 languages to maybe request some other context information as well.
601 
602 Having said this, some apriori guidelines on contexts can be followed.
603 
604 Since in English the form of an adjective does not change based on
605 the gender of the noun it modifies, properly translating messages which
606 are single standalone adjectives will be impossible in many languages
607 without a context. So, in general, every message which is a standalone
608 adjective should have context:
609 
610 ~~~
611 i18nc("new file", "New");
612 i18nc("default action", "Default");
613 ~~~
614 
615 
616 Lists of related items typically benefit from having the same context,
617 since they should all be translated in the same style:
618 
619 ~~~
620 i18nc("IPv4 configuration method", "Automatic (VPN)");
621 i18nc("IPv4 configuration method", "Automatic (VPN) addresses only");
622 i18nc("IPv4 configuration method", "Automatic (PPP)");
623 i18nc("IPv4 configuration method", "Automatic (PPP) addresses only");
624 ...
625 ~~~
626 
627 
628 When there are placeholders in the text for which it is not clear,
629 from the text alone, what kind of argument they represent,
630 this should be explained in the context:
631 
632 ~~~
633 i18nc("%1 old file path, %2 new file path",
634  "Cannot move '%1' to '%2'.", oldPath, newPath);
635 i18nc("%1 track URL, %2 artist name",
636  "%1 (by %2)", trackUrl, artistName);
637 ~~~
638 
639 
640 It is frequently suggested to state in the context the grammar category
641 of the message text, such as "this is a verb" or "this is a noun".
642 Since the grammar category of the translation does not have to be
643 the same as that of the original, this kind of context provides
644 circumstantial information at best (see the section \ref uimark_ctxt
645 for what translators may use it to draw some real information about),
646 and is worthless at worst.
647 Also, due to relative absence of declension in English grammar,
648 different programmers may have different opinion on the grammar category:
649 the menu title "View", is it a verb or a noun?

654

656 for the programmers to follow, translators should establish HIG-like
657 convention for their language concerning the forms of UI text.
658 Therefore, for a proper translation, the translator will need too know

811 <a name="ctxt_extc">
812 
813 #### Disambiguation Context vs. Extracted Comment
814 
815 The GNU Gettext system actually defines *two* types of context
816 for translators. The type discussed so far, the first argument
817 of `*i18nc*` calls, is called *disambiguation context*.
818 The other type of context is *extracted comment*.
819 In Ki18n, this other type of context is written as a code comment,
820 in the line just preceding the message text,
821 and starting with *i18n:* keyword:
822 
823 ~~~
824 // i18n: This is a short text containing all letter of the alphabet,
825 // e.g. for testing fonts. Translate with a sentence which can
826 // serve the same purpose in your language.
827 i18n("The quick brown fox jumps over the lazy dog");
828 ~~~
829 
830 
831 There are two main differences between disambiguation contexts and
832 extracted comments.
833 
834 The first difference is that extracted comments
835 *do not separate messages* in the translation catalog.
836 For example, such two messages equipped with extracted comments:
837 
838 ~~~
839 // i18n: new file
840 QString msg1 = i18n("New");
841 // i18n: new folder
842 QString msg2 = i18n("New");
843 ~~~
844 
845 will show up in the translation catalog as a single message
846 with aggregate comments:
847 
848 ~~~
849 #. i18n: new file
850 #. i18n: new folder
851 msgid "New"
852 msgstr ""
853 ~~~
854 
855 The same two messages equipped with disambiguation contexts:
856 
857 ~~~
858 QString msg1 = i18nc("new file", "New");
859 QString msg2 = i18nc("new folder", "New");
860 ~~~
861 
862 will show up in the translation catalog as two messages:
863 
864 ~~~
865 msgctxt "new file"
866 msgid "New"
867 msgstr ""
868 
869 msgctxt "new folder"
870 msgid "New"
871 msgstr ""
872 ~~~
873 
874 
875 The second difference is that a change in extracted comment
876 *does not invalidate the existing translation*, i.e.
877 it will not force translators to revisit the message and revise
878 the translation (see the section \ref handle_update for details
879 on how this invalidation happens with disambiguation contexts).
880 Thus, for example, if there is a "message freeze"
881 before the next release of a piece of software, during which
882 programmers must not modify messages so that translators can
883 thoroughly complete their work, it is allowed to modify
884 the extracted comment, but it is not allowed to modify
885 the disambiguation context.
886 
887 The Gettext manual suggests to use extracted comments as
888 the default way of providing context for translators,
889 and to use disambiguation contexts only when message separation
890 is necessary. However, we (the people in the KDE community)
891 suggest the opposite -- to use disambiguation context by default,
892 and extracted comments only in special cases.
893 This because there are two dangers associated with extracted comments:
894 one is that programmer may fail to properly judge when two messages
895 should be made separate for translation (and having to judge that
896 all the time is a burden in the first place), and the other is that
897 when the context change is such that translators really should revisit
898 the message, they would not get any automatic signal about that.
899 The message freeze advantage of extracted comments has not been
900 observed to be very important.
901 
902 One special case when to an use extracted comment is when the context
903 is a multi-sentence text explaining the purpose of the message,
904 and the context is unlikely to change (in its meaning).
905 Another case is when the context lists a fixed set
906 of possible translations ("meta-messages", by which translators
907 influence some aspect of text formatting), which may expand in
908 the future, as new possibilities are introduced at request of
909 translators into new languages.

911 <a name="i18n_noop">
912 
913 ### Extraction-Only Macros
914 
915 Sometimes it is convenient only to mark a message for extraction
916 (into the catalog template, as described in \ref handle_extract),
917 and to place the actual `i18n` call somewhere else.
918 A typical case of this are global static initializers,
919 where only POD types can be safely used.
920 For this purpose `I18N*_NOOP` macros are provided.
921 
922 The `I18N_NOOP` macro is the counterpart to `*i18n` calls,
923 and it is used like this:
924 
925 ~~~
926 typedef struct
927 {
928  const char *description;
929  const char *regExp;
930 } term;
931 // Inert I18N_NOOP macros here...
932 static const term items[] = {
933  {I18N_NOOP("any character"), "."},
934  {I18N_NOOP("start of line"), "^"},
935  {I18N_NOOP("end of line"), "$"},
936  {I18N_NOOP("repeats, zero or more times"), "*"},
937  ...
938 };
939 
940 ...
941 
942 void populatePatternMenu (QMenu *menu)
943 {
944  for (uint i = 0; i < sizeof(items) / sizeof(items[0]); i++) {
945  menu->addAction(new RegExpAction(menu,
946  // ...actual i18n call here.
947  i18n(items[i].description),
948  items[i].regExp));
949  }
950 }
951 ~~~
952 
953 
954 The `I18NC_NOOP` macro is the counterpart to `*i18nc` calls:
955 
956 ~~~
957 typedef struct
958 {
959  const char *description, *abbrev;
960 } unitDef;
961 static const unitDef units[] = {
962  {I18N_NOOP2_NOSTRIP("unit for 2^10 bytes", "KiB")},
963  {I18N_NOOP2_NOSTRIP("unit for 2^20 bytes", "MiB")},
964  ...
965 }
966 ...
967  QString unitAbbrev = i18nc(units[i].description, units[i].abbrev);
968 ~~~
969 
970 
971 There are also two deprecated macros:
972 `I18N_NOOP2_NOSTRIP` is simply the old name of `I18NC_NOOP`;
973 `I18N_NOOP2` takes the context argument but discards it,
974 which means that the context must be repeated verbatim in
975 the corresponding `i18nc` call.
976 
977 In general, `I18N*_NOOP` macros make it harder to follow
978 i18n in the code, and should be avoided when possible.

981 <a name="link_cat">
982 
983 ## Connecting Calls to Catalogs
984 
985 Every `i18n` call must look for translations in exactly
986 one translation catalog for a given language of translation.
987 For this purpose, a group of catalogs which have the same source text
988 but translations into different languages, is identified by
989 a unique canonical name, called the *domain*.
990 Therefore, every `i18n` call must be connected to a domain.
991 This connection is established differently for applications
992 and libraries, though the difference is only for convenience: if desired,
993 the more verbose library method can be used for application code as well.
994 
995 <a name="link_prog">
996 
997 ### Connecting to Catalogs in Application Code
998 
999 All `*i18n*` calls in an application can be connected
1000 to a single domain by calling the static
1001 `KLocalizedString::setApplicationDomain` method with
1002 the domain as the argument:
1003 
1004 ~~~
1005 #include <klocalizedstring.h>
1006 
1007 ...
1008 
1009 int main (int argc, char *argv[])
1010 {
1011  ...
1012  KLocalizedString::setApplicationDomain("fooapp");
1013  ...
1014 }
1015 ~~~
1016 
1017 This call can be made at any place in the code
1018 (e.g. there is no need to have `QApplication` constructed beforehand),
1019 but of course before any `i18n` call takes place.
1020 
1021 This is all there is to connecting calls and catalogs application's
1022 C++ source files. However, there may also be some non-code files
1023 that need connecting, and how to do this is some typical non-code
1024 files is described in the section \ref link_noncode.
1025 
1026 <a name="link_lib">
1027 
1028 ### Connecting to Catalogs in Library Code
1029 
1030 `i18n` calls in libraries must be strictly connected to library's
1031 own translation domain no matter how the library is used,
1032 and this should be fully transparent to the library's client.
1033 In particular, if the client does not use Ki18n for internationalization
1034 of its own messages, library translation must still work as expected.
1035 Therefore, in library code, the call-domain connection is established
1036 in this way:
1037 
1038 ~~~
1039 #define TRANSLATION_DOMAIN "foolib"
1040 #include <klocalizedstring.h>
1041 
1042 ...
1043 
1044 void some_library_function ()
1045 {
1046  ...
1047  QString msg = i18n("Greetings from Foolib!");
1048  ...
1049 }
1050 ~~~
1051 
1052 The definition of `TRANSLATION_DOMAIN` triggers
1053 the domain-specialization macro in `klocalizedstring.h`.
1054 It routes all `*i18n*` calls to their `*i18nd*` counterparts,
1055 which take the domain as their first argument.
1056 The `i18n` call from this example will thus expand into:
1057 
1058 ~~~
1059 QString msg = i18nd("foolib", "Greetings from Foolib!");
1060 ~~~
1061 
1062 
1063 It is possible to use `*i18nd*` calls explicitly,
1064 but there should be no need for that.
1065 If there are any messages that should draw translations
1066 from a special domain, it is better style-wise to use the
1067 `ki18n("...").toString(domain)` construct.
1068 
1069 Definition of `TRANSLATION_DOMAIN` can be put into
1070 a private header file of the library, so that it does not have
1071 to be repeated at multiple locations.
1072 If there are some `i18n` calls in a *public* header file,
1073 definition of `TRANSLATION_DOMAIN` would propagate
1074 into and affect the application client code that uses Ki18n too.
1075 This is prevented by adding the following lines somewhere
1076 after the last `i18n` call in the public header:
1077 
1078 ~~~
1079 #undef TRANSLATION_DOMAIN
1080 #include <klocalizedstring.h>
1081 ~~~
1082 
1083 This will undefine all expansions of `*i18n*` into `*i18nd*`,
1084 leaving the client's environment clean.
1085 If instead the public header contains only `I18N_NOOP*` macros,
1086 defining `TRANSLATION_DOMAIN` is unnecessary in the first place,
1087 since actual `i18n` calls happen somewhere else.
1088 
1089 <a name="link_noncode">
1090 
1091 ### Connecting to Catalogs in Non-Code Files
1092 
1093 Both KDE applications and libraries can include some non-code files
1094 which contain messages that need to be connected to a translation domain.
1095 This can be the same domain where messages from C++ code are found,
1096 or another domain, whatever seems more appropriate.
1097 In principle, each type of non-code file requires its own connection
1098 mechanism, and here it is explained how this works for
1099 typical types of non-code files found in KDE sources.
1100 
1101 It is assumed in the following that all messages in the non-code file
1102 are connected to the single domain. In other words, the connection
1103 is specified on the file level rather than on the message level.
1104 
1105 <a name="link_ui">
1106 
1107 #### Qt Designer (.ui) files
1108 
1109 First, to have UI strings from `.ui` file passed through Ki18n,
1110 `uic` is run with `-tr tr2i18n`. This will replace all
1111 native Qt `tr` calls with Ki18n's `tr2i18n` calls
1112 in the resulting header file.
1113 Then, the generated header file needs to be post-processed
1114 to fix empty messages and include `klocalizedstring.h`.
1115 At this point, the `TRANSLATION_DOMAIN` can be defined just like
1116 in static C++ files.
1117 
1118 If CMake is used as the build system, a macro that performs
1119 all of the above is provided (`ki18n_wrap_ui`).
1120 Otherwise, one could use a shell snippet such as this:
1121 
1122 ~~~
1123 domain=fooapp
1124 uifile=fooconfpage.ui
1125 uihfile=$uifile.h
1126 uic -tr tr2i18n $uifile -o $uihfile
1127 sed -i 's/tr2i18n("")/QString::null/g' $uihfile
1128 sed -i 's/tr2i18n("", "")/QString::null/g' $uihfile
1129 sed -i "1i\#define TRANSLATION_DOMAIN \"$domain\"\n#include <klocalizedstring.h>" $uihfile
1130 ~~~
1131 
1132 
1133 If strings contain KUIT markup (section \ref kuit_markup),
1134 `tr2i18n` in the lines above should be replaced with `tr2xi18n`.

1136 <a name="link_rc">
1137 
1138 #### KXmlGui (.rc) files
1139 
1140 Since `.rc` files are interpreted at runtime,
1141 the translation domain connection is established simply
1142 by adding the `translationDomain` attribute to the top element:
1143 
1144 ~~~
1145 <!DOCTYPE kpartgui SYSTEM "kpartgui.dtd">
1146 <kpartgui name="foolib_part" version="55" translationDomain="foolib">
1147 ...
1148 ~~~
1149 
1150 If the `.rc` file belongs to application rather than library source,
1151 it is not necessary to set `translationDomain`. If not set,
1152 translations will be looked up in the domain set with
1153 `KLocalizedString::setApplicationDomain` call in the code.
1154 
1155 If strings contain KUIT markup (section \ref kuit_markup),
1156 additionally the attribute `translationMarkup="true"` should be set.

1158 <a name="link_kcfg">
1159 
1160 #### KConfigXT (.kcfg) files
1161 
1162 Instructions for building the configuration code from a `.kcfg` file
1163 are contained in the `.kcfgc` file of the same base name;
1164 `kconfig_compiler` is invoked with both files as arguments.
1165 Then, the domain connection is established simply by adding
1166 the `TranslationSystem` and `TranslationDomain` fields in
1167 the `.kcfgc` file, to select Ki18n as the translation system
1168 and the appropriate translation domain:
1169 
1170 ~~~
1171 File=foolib.kcfg
1172 ...
1173 TranslationSystem=kde
1174 TranslationDomain=foolib
1175 ~~~
1176 
1177 If the `.kcfg` file is part of an application rather than a library,
1178 the `TranslationDomain` field can be omitted in order
1179 to have messages looked up in the domain set by
1180 `KLocalizedString::setApplicationDomain` call in the code.
1181 
1182 If strings contain KUIT markup (section \ref kuit_markup),
1183 additionaly the field `TranslationMarkup=true` should be set.

1185 <a name="handle_cat">
1186 
1187 ## Handling Catalog Files
1188 
1189 For translators to start working, one or more translation catalog files
1190 should be prepared, based on the `i18n` calls in the source code.
1191 The procedure to do this is called *extraction* of messages.
1192 Extraction produces empty catalog files, called *templates*.
1193 These files are in the PO format, and have `.pot` extension.
1194 Section \ref handle_extract explains how to perform extraction.
1195 
1196 Once templates are ready, a translators make copies of them,
1197 with `.po` extension, and start filling them out with translations
1198 into respective languages.
1199 When translation is done, the translated catalog is committed
1200 into the source code repository.
1201 The build system is set up to install translated catalogs.
1202 Section \ref handle_install provides necessary steps for this.
1203 
1204 After some development has passed, the source repository will
1205 contain many translated catalogs which are out of date with
1206 respect to latest catalog templates.
1207 Of course, translators do not have to start translating from scratch,
1208 but there are specialized tools to carry over as much of existing
1209 translation as possible, so that only new and modified texts
1210 need to be considered.
1211 Section \ref handle_update shows how this is done.
1212 
1213 A usual application or a library has one translation catalog,
1214 but there can be more if there is higher modularity of the source.
1215 The following subsections refer to a single catalog wherever
1216 the extension to case with multiple catalogs is obvious,
1217 and mention multiple catalogs only where necessary.

1219 <a name="handle_extract">
1220 
1221 ### Extracting Templates
1222 
1223 The primary tool for collecting texts from `i18n` calls and
1224 writing out the catalog template is `xgettext`,
1225 from the official Gettext tools package.
1226 `xgettext` supports many programming languages and sublanguage
1227 environments, among which naturally C++ and Ki18n specifically.
1228 
1229 The extraction process from source code files is thus simple:
1230 `xgettext` runs with appropriate options over all files,
1231 and it writes out the catalog template.
1232 For the moment masking the complete list of options as `$EXTOPTS`,
1233 `xgettext` can be run for example like this at the top of Fooapp
1234 source to create the catalog template `fooapp.pot`:
1235 
1236 ~~~
1237 find -name \*.cpp -o -name \*.h -o -name \*.qml | sort \
1238 | xargs xgettext $EXTOPTS -o fooapp.pot
1239 ~~~
1240 
1241 Or, a list of source files that should be extracted from can be
1242 assembled separately and fed to `xgettext:`
1243 
1244 ~~~
1245 # ...create sources.list...
1246 xgettext $EXTOPTS -f sources.list -o fooapp.pot
1247 ~~~
1248 
1249 One may want to assemble the list of source files by hand
1250 or semi-automatically in order to prioritize the order of translation
1251 (messages from most important files appearing first in the catalog),
1252 to exclude some portions of the source tree from extraction, and so on.
1253 
1254 `$EXTOPTS` that cover everything from Ki18n, and some generalities,
1255 should look like this:
1256 
1257 ~~~
1258 --c++ --kde \
1259 --from-code=UTF-8 \
1260 -c i18n \
1261 -ki18n:1 -ki18nc:1c,2 -ki18np:1,2 -ki18ncp:1c,2,3 \
1262 -kki18n:1 -kki18nc:1c,2 -kki18np:1,2 -kki18ncp:1c,2,3 \
1263 -kI18N_NOOP:1 -kI18NC_NOOP:1c,2 \
1264 --copyright-holder=<author-of-original-text> \
1265 --msgid-bugs-address=<where-to-report-errors-in-original-text>
1266 ~~~
1267 
1268 `--c++ --kde` options tell `xgettext` that source files
1269 are C++ with Ki18n. `--from-code=UTF-8` specifies the encoding
1270 of source files to be UTF-8, which must be so for Ki18n.
1271 `-c i18n` states that comments for extraction start with given
1272 keyword (`// i18n: ...`).
1273 The series of `-k` options informs `xgettext` of all possible
1274 translation call names and which of their arguments to extract.
1275 Finally, options `--copyright-holder` and
1276 `--msgid-bugs-address` automatically write
1277 the corresponding information into the catalog at proper place.
1278 If there are semantic markup calls in the code (section \ref kuit_markup),
1279 the following `-k` options should be added as well:
1280 
1281 ~~~
1282 -kxi18n:1 -kxi18nc:1c,2 -kxi18np:1,2 -kxi18ncp:1c,2,3 \
1283 -kkxi18n:1 -kkxi18nc:1c,2 -kkxi18np:1,2 -kkxi18ncp:1c,2,3 \
1284 ~~~
1285 
1286 
1287 `xgettext` unfortunately cannot be directly used to extract messages
1288 from the usual XML files appearing in Qt and KDE sources --
1289 Designer (`.ui`), KXmlGui (`.rc`) and KConfigXT (`.kcfg`) files.
1290 Therefore the `kdesdk` package provides the `extractrc` script,
1291 which extracts XML messages as dummy `i18n` calls into a dummy C++ file,
1292 This file can then be included into the list of files for `xgettext` run.
1293 The usual invocation of `extractrc` is:
1294 
1295 ~~~
1296 find -name \*.ui -o -name \*.rc -o -name \*.kcfg | sort \
1297 | extractrc > rc.cpp
1298 # ...run xgettext with rc.cpp included in source files...
1299 rm rc.cpp
1300 ~~~
1301 
1302 
1303 If the catalog being extracted is an application catalog,
1304 i.e. given as `KLocalizedString::setApplicationDomain` in the code,
1305 it should contain two meta-messages for translation credits
1306 which will be shown by `KAboutApplicationDialog`.
1307 These messages are also written as dummy `i18n` calls,
1308 usually into the same dummy C++ file with XML messages,
1309 with the following context and text:
1310 
1311 ~~~
1312 echo 'i18nc("NAME OF TRANSLATORS", "Your names");' >> rc.cpp
1313 echo 'i18nc("EMAIL OF TRANSLATORS", "Your emails");' >> rc.cpp
1314 ~~~
1315 
1316 
1317 The extraction command sequence can be written down as a small script,
1318 to be run periodically by the maintainer,
1319 or it can be integrated into the build system.
1320 
1321 For the code residing in the official KDE repositories
1322 a special form of the extraction script is mandated.
1323 This enables automatic overnight template extraction and feeding
1324 into dedicated translation section of KDE repositories.
1325 Details can be found at
1326 <a href="http://techbase.kde.org/Development/Tutorials/Localization/i18n_Build_Systems">KDE Techbase</a>.

1329 
1330 ### Placing and Installing Catalogs
1331 
1332 For an application or a library which uses a single catalog,
1333 the usual organization of catalogs in the source tree is this:
1334 
1335 ~~~
1336 fooapp/
1337  src/
1338  doc/
1339  po/
1340  fooapp.pot
1341  aa.po
1342  bb.po
1343  cc.po
1344  ...
1345 ~~~
1346 
1347 Here translated catalog files are named by their language codes
1348 (`aa`, `bb`, `cc`...).
1349 In case of multiple catalogs, one directory per catalog can be created
1350 under the `po/` directory:
1351 
1352 ~~~
1353 fooapp/
1354  src/
1355  lib
1356  doc/
1357  po/
1358  fooapp/
1359  fooapp.pot
1360  aa.po
1361  bb.po
1362  ...
1363  foolib/
1364  foolib.pot
1365  aa.po
1366  bb.po
1367  ...
1368 ~~~
1369 
1370 An alternative organization is to have one directory per language,
1371 and name catalog files by the translation domain.
1372 In multiple catalog situation this would look like:
1373 
1374 ~~~
1375 fooapp/
1376  src/
1377  lib
1378  doc/
1379  po/
1380  templates/
1381  fooapp.pot
1382  foolib.pot
1383  aa/
1384  fooapp.po
1385  foolib.po
1386  bb/
1387  fooapp.po
1388  foolib.po
1389  ...
1390 ~~~
1391 
1392 
1393 Catalog templates are fully derived files, and therefore
1394 some maintainers do not like to keep them inside the repository.
1395 In that case, at least the tarball should contain the templates
1396 (i.e. they should be generated at packaging time),
1397 so that translators have somewhere to get them from.
1398 Another possibility is to upload the template to
1399 a *translation hub* (such as
1400 <a href="https://www.transifex.com/">Transifex</a>),
1401 which translators can use to upload translated catalogs back,
1402 and usually for some other features as well (assignment, review, etc).
1403 
1404 If the code resides in an official KDE repository, neither templates
1405 nor translated catalogs are kept inside the source tree.
1406 Instead, translated catalogs are fetched from an appropriate place
1407 when tarball is made, using a script provided for that purpose.
1408 Details can be found at
1409 <a href="http://techbase.kde.org/Development/Tutorials/Localization/i18n_Build_Systems">KDE Techbase</a>.
1410 
1411 No matter how the catalog files are named and organized inside
1412 the distribution tarball, they must be installed in exactly one way.
1413 The base name of the installed catalog must be their translation domain,
1414 and if the package is installed into `$PREFIX`, the installed directory
1415 tree must look like as follows:
1416 
1417 ~~~
1418 $PREFIX/
1419  share/
1420  locales/
1421  aa/
1422  LC_MESSAGES/
1423  fooapp.mo
1424  foolib.mo
1425  bb/
1426  LC_MESSAGES/
1427  fooapp.mo
1428  foolib.mo
1429  ...
1430 ~~~
1431 
1432 Given that these directories are shared with other packages
1433 in the same prefix, by Gettext convention translation domains
1434 must be unique (like package names are).
1435 
1436 MO files are the compiled version of PO files, and they are produced
1437 using Gettext's `msgfmt` command:
1438 
1439 ~~~
1440 msgfmt $SRCDIR/po/fooapp/aa.po -o $BUILDDIR/mo/aa/fooapp.mo
1441 ~~~
1442 
1443 Compilation and installation of catalogs should naturally be integrated
1444 into the build system. In case <a href="http://www.cmake.org/">CMake</a>
1445 is used, KDE provides CMake macros for this purpose.

1450 
1451 Since Ki18n provides a run-time scripting capability for translators,
1452 some translators may also write the scripting module corresponding
1453 to the catalog.
1454 A scripting module is a directory named like the translation domain,
1455 and at least one JavaScript file inside, also with the same base name
1456 as the translation domain.
1457 
1458 Scripting modules can be placed like this in the source tree:
1459 
1460 ~~~
1461 fooapp/
1462  po/
1463  fooapp.pot
1464  aa.po
1465  aa/
1466  fooapp.js
1467  ...
1468  bb.po
1469  ...
1470 ~~~
1471 
1472 or in the per-language directory variant:
1473 
1474 ~~~
1475 fooapp/
1476  po/
1477  templates/
1478  fooapp.pot
1479  aa/
1480  fooapp.po
1481  fooapp/
1482  fooapp.js
1483  ...
1484  bb/
1485  fooapp.po
1486  ...
1487 ~~~
1488 
1489 
1490 The installation location for scripting modules is like that for catalogs,
1491 only using `LC_SCRIPTS/` directory instead of `LC_MESSAGES/`:
1492 
1493 ~~~
1494 $PREFIX/
1495  share/
1496  locales/
1497  aa/
1498  LC_SCRIPTS/
1499  fooapp/
1500  fooapp.js
1501  ...
1502 ~~~
1503 
1504 
1505 When a translator inquires about adding a scripting module,
1506 or sends one in, the maintainer should check with the translator
1507 if perhaps the functions provided by the module are more widely
1508 applicable. If that is the case, they should rather become part of
1509 Ki18n's own scripting module, because then they will be accessible
1510 to all Ki18n-based translations in the given language.

1512 <a name="handle_update">
1513 
1514 ### Updating Catalogs
1515 
1516 When new catalog template is extracted after some development
1517 has been done, existing translation should be updated against it.
1518 This is called *merging* with template, and it is performed
1519 by Gettext's `msgmerge` command.
1520 There are some merging options that can be examined here,
1521 but generally the best invocation of `msgmerge` is this:
1522 
1523 ~~~
1524 msgmerge --update --backup=none --previous aa.po fooapp.pot
1525 ~~~
1526 
1527 Options `--update --backup=none` mean to update the catalog
1528 in place, and not to make a backup file. Option `--previous`
1529 puts some additional information into every modified message,
1530 that translation editing tools can use to show to the translator
1531 all changes in the text.
1532 This command should be run once for every existing translation catalog.
1533 
1534 One thing to keep in mind is that a change in the context string
1535 of a message in the code (i.e. the first argument to `*i18nc*` calls),
1536 including adding or removing one, will also register as
1537 a modified message in the merged catalog.
1538 This will require that translators revisit it, which is exactly as
1539 intended: if the context has changed, especially if it was added,
1540 some changes in translation may be needed.
1541 However, this means that when a "message freeze" is declared so that
1542 translators can complete updating translations without disruption,
1543 contexts fall under same rules as text.
1544 
1545 There are a few possibilities for who and when should perform merging.
1546 For example, the maintainer can write a script that at the same time
1547 extracts the template and merges all catalogs, and run it periodically,
1548 committing updated catalogs to the repository for translators to pick up.
1549 This could even be integrated into the build system.
1550 Some maintainers do not like committing automatic changes,
1551 and instead expect translators to run the extraction-merging script
1552 for the language they maintain, update the translation,
1553 and commit only that updated catalog. This solution is cleaner with
1554 respect to repository history, but it may burden translators.
1555 
1556 When operating in an official KDE repository, maintainers do not
1557 have to deal with merging at all. The server-side automation which
1558 automatically extracts templates and provides them to translators,
1559 also performs merging. So the maintainer is left only to pick up
1560 whatever are the latest catalogs when making a tarball.

1563 <a name="kuit_markup">
1564 
1565 ## Semantic Markup
1566 
1567 When composing user-interface text, some programmers ponder about
1568 the typographical conventions to use in certain contexts.
1569 For example, when a file name is inserted into the text,
1570 some typographical solutions that can be used are:
1571 
1572 ~~~
1573 i18n("Cannot open %1 for reading.", filePath);
1574 i18n("Cannot open '%1' for reading.", filePath);
1575 i18n("Cannot open \"%1\" for reading.", filePath);
1576 ~~~
1577 
1578 For the Qt widgets that have rich text capability,
1579 exposed as subset of HTML tags, additional solutions include:
1580 
1581 ~~~
1582 i18n("Cannot open <b>%1</b> for reading.", filePath);
1583 i18n("Cannot open <i>%1</i> for reading.", filePath);
1584 i18n("Cannot open <tt>%1</tt> for reading.", filePath);
1585 ~~~
1586 
1587 The problem here is not so much to decide on one solution, as it is
1588 to follow it consistently through time and between contributors.
1589 One may also want to use two solutions, one in places where only
1590 plain text is allowed, and another where rich text is available.
1591 Wouldn't it then be easier to write everywhere:
1592 
1593 ~~~
1594 xi18n("Cannot open <filename>%1</filename> for reading.", filePath);
1595 ~~~
1596 
1597 and have it automatically resolve into plain or rich text according to
1598 the UI context, using formatting patterns defined at one place?
1599 This approach is called *semantic* markup, because the author
1600 marks parts of the text according to what they *represent*.
1601 
1602 Ki18n implements such a semantic markup,
1603 called KUIT (KDE User Interface Text).
1604 It is accessed through the series of `xi18n*` and `kxi18n*` calls,
1605 which are the KUIT-aware counterparts of `i18n*` and `ki18n*` calls.
1606 Ordinary and KUIT-aware calls can be freely mixed within
1607 a given body of code.
1608 
1609 KUIT defines a number of semantic tags that are frequently of use,
1610 as listed in the section \ref kuit_tags.
1611 But, KUIT also allows the programmer to define custom tags,
1612 as well as to change visual formatting patterns for predefined tags.
1613 This capability should lessen one important issue of semantic markups:
1614 when the author is forced to choose between several tags
1615 none of which exactly fits the desired meaning;
1616 with KUIT, the author can simply define a custom tag in that case.
1617 
1618 <a name="kuit_def_tags">
1619 
1620 ### Defining Tags
1621 
1622 Tags are defined and redefined per translation domain, so that changes
1623 will not affect markup resolution in any other domain.
1624 Changes are performed through the `KuitSetup` object associated
1625 with the domain, as returned by the `Kuit::setupForDomain` method.
1626 A tag is defined (or redefined) by defining (or redefining)
1627 its formatting patterns, with one call to `KuitSetup::setTagPattern`
1628 for each desired combination of tag name, attribute names,
1629 and visual format.
1630 Here is an example of defining the tag `<player>`,
1631 which has an optional attribute `color=`,
1632 on the domain `foogame`:
1633 
1634 ~~~
1635 KuitSetup *ks = Kuit::setupForDomain("foogame");
1636 QString tagName;
1637 QStringList attribNames;
1638 
1639 tagName = "player";
1640 attribNames.clear()
1641 ks->setTagPattern(tagName, attribNames, Kuit::PlainText,
1642  ki18nc("tag-format-pattern <player> plain",
1643  "'%1'"));
1644 ks->setTagPattern(tagName, attribNames, Kuit::RichText,
1645  ki18nc("tag-format-pattern <player> rich",
1646  "<b>%1</b>"));
1647 attribNames.append("color");
1648 ks->setTagPattern(tagName, attribNames, Kuit::RichText,
1649  ki18nc("tag-format-pattern <player color= > rich",
1650  "<font color='%2'><b>%1</b></font>"));
1651 ~~~
1652 
1653 The first two `setTagPattern` calls set up resolution of
1654 `<player>` without attributes, into plain and rich text.
1655 The third call sets up resolution for `<player color="...">`,
1656 but only into rich text; since a plain text pattern is not defined
1657 for this tag-attribute combination, it will fall back to basic
1658 `<player>` plain text pattern.
1659 A fallback is always defined, the elementary fallback being a no-op,
1660 where tag is simply removed.
1661 Formatting patterns must be wrapped for translation too,
1662 since translators may need to tweak them;
1663 *ordinary* (not markup-aware) `ki18nc` calls must be used here,
1664 since patterns themselves are not KUIT markup.
1665 The `%1` placeholder in the pattern will be replaced by
1666 the text wrapped with the tag, and `%2` and upwards with
1667 attribute values, in the order of appearance in attribute names list.
1668 
1669 If a simple substitution pattern is insufficient for formatting,
1670 additionally a formatting function of type `Kuit::TagFormatter`
1671 can be given. The result of this function is substituted into the pattern;
1672 alternatively an empty pattern can be given (as `KLocalizedString()`),
1673 in which case the result is used directly, no substitution is performed.
1674 The formatting function also receives the current element path,
1675 so that the resolution can depend on the markup tree context if needed.
1676 Anything in the function that may need translator input
1677 should be appropriately exposed through `i18nc*` or `ki18nc*` calls.
1678 
1679 In the section \ref kuit_tags it is stated that every KUIT tag
1680 is classified either as a phrase tag or as a structuring tag,
1681 and explained what that means for processing.
1682 A newly defined tag is by default a phrase tag;
1683 method `KuitSetup::setTagClass` can be used to change its class:
1684 
1685 ~~~
1686 tagName = "list2";
1687 ks->setTagPattern(tagName, ...);
1688 ks->setTagClass(tagName, Kuit::StructTag);
1689 ~~~
1690 
1691 
1692 In a library, changes to the KUIT setup may need to be private,
1693 applicable only in library's own domain, but they may also need
1694 to be public, applicable in clients' domains.
1695 For changes that should be public, the library should define
1696 a public function which takes the domain as the argument
1697 and performs all the changes on that domain:
1698 
1699 ~~~
1700 void updateKuitSetup (const char *domain)
1701 {
1702  KuitSetup *ks = Kuit::setupForDomain(domain);
1703  ....
1704 }
1705 ~~~
1706 
1707 The client code should then call this function in its initialization.

1710 
1711 ### Selecting Visual Format
1712 
1713 The target visual format for any given `xi18n` call can be selected
1714 in two ways.
1715 
1716 The primary way is by UI markers in the message context,
1717 which were described in the section \ref uimark_ctxt.
1718 Every `@<major>:<minor>` marker combination has
1719 a default target visual format assigned, as follows:
1720 <table>
1721 <tr><th>UI Marker</th>
1722  <th>Visual Format</th></tr>
1723 <tr><td>(none)</td>
1724  <td>plain</td></tr>
1725 <tr><td>\@action:<any></td>
1726  <td>plain</td></tr>
1727 <tr><td>\@title:<any></td>
1728  <td>plain</td></tr>
1729 <tr><td>\@option:<any></td>
1730  <td>plain</td></tr>
1731 <tr><td>\@label:<any></td>
1732  <td>plain</td></tr>
1733 <tr><td>\@item:<any></td>
1734  <td>plain</td></tr>
1735 <tr><td>\@info, \@info:tooltip, \@info:whatsthis, \@info:usagetip</td>
1736  <td>rich</td></tr>
1737 <tr><td>\@info:status, \@info:progress, \@info:credit</td>
1738  <td>plain</td></tr>
1739 <tr><td>\@info:shell</td>
1740  <td>term</td></tr>
1741 </tr>
1742 </table>
1743 Target visual formats associated with UI markers can be changed using
1744 the `KUITSetup::setFormatForMarker` method:
1745 
1746 ~~~
1747 KuitSetup *ks = Kuit::setupForDomain("fooapp");
1748 // Set standalone @info (no minor component) to plain text:
1749 ks->setFormatForMarker("@info", Kuit::PlainText);
1750 // Set @info:tooltip to rich text:
1751 ks->setFormatForMarker("@info:tooltip", Kuit::RichText);
1752 // Set all @info:<minor> to plain text:
1753 ks->setFormatForMarker("@info:", Kuit::PlainText);
1754 ~~~
1755 
1756 
1757 The second way to select the visual format is by using a `kxi18n*` call,
1758 and passing the format type to the `toString` method:
1759 
1760 ~~~
1761 kxi18n("@info", "Logging paused.").toString(Kuit::PlainText);
1762 ~~~
1763 
1764 This will override any format implied by the UI marker if present.
1765 
1766 If a library is making modifications in visual format association
1767 with UI markers, and these changes should be available to clients,
1768 the same approach as in the section \ref kuit_def_tags should be used.

1770 <a name="kuit_escape">
1771 
1772 ### Escaping
1773 
1774 While for `*i18n*` calls it was advised to keep each message text
1775 well-formed by itself with respect to Qt rich text markup,
1776 for `*xi18n*` calls well-formedness is mandatory.
1777 This means that markup-significant characters in plain-looking text
1778 need to be escaped, using standard XML entities:
1779 
1780 ~~~
1781 xi18n("Installed Fooapp too old, need release &gt;= 2.1.8.");
1782 xi18n("Set &lt;themeId&gt; as theme on startup");
1783 ~~~
1784 
1785 The exception is the ampersand (&) character, which in XML denotes
1786 the start of an entity, but in Qt denotes the accelerator marker.
1787 Therefore it is necessary to escape ampersand as `&amp;`
1788 only when it is in position which would result in valid entity syntax:
1789 
1790 ~~~
1791 // Escaping not needed because not in entity-like position:
1792 xi18n("Remove &all entries");
1793 // Escaping not needed for the same reason wrt. markup,
1794 // but ampersand doubled to escape it wrt. Qt accelerator marker:
1795 xi18n("Look && Feel");
1796 // Escaping wrt. markup necessary:
1797 xi18n("Delete &amp;everything; see if I care!");
1798 ~~~
1799 
1800 The example for necessary escaping above is rather artificial,
1801 because in practice it is unlikely for ampersand to appear
1802 in entity-like position while not actually starting an entity.
1803 
1804 To assure the validity of markup when arguments are inserted into
1805 `xi18n`-wrapped text, all markup-significant characters
1806 in string arguments are automatically escaped.
1807 For example, this works as expected:
1808 
1809 ~~~
1810 QString filePath("assignment03-<yourname>.txt");
1811 QString msg1 = kxi18n("Delete <filename>%1</filename>?", filePath)
1812  .toString(Kuit::PlainText);
1813 // msg1 == "Delete 'assignment03-<yourname>.txt'?"
1814 QString msg2 = kxi18n("Delete <filename>%1</filename>?", filePath)
1815  .toString(Kuit::RichText);
1816 // msg2 == "Delete `assignment03-&lt;yourname&gt;.txt`?"
1817 ~~~
1818 
1819 But, how then to compose a text where the arguments too should
1820 contain some KUIT markup? This is done by using non-finalization
1821 `kxi18n*` call to translate the argument text, and passing
1822 the returned `KLocalizedString` object directly as argument:
1823 
1824 ~~~
1825 KLocalizedString stateDesc = kxi18nc(
1826  "On <emphasis>indefinite</emphasis> hold.");
1827 QString msg = xi18n("@info",
1828  "<para>Task state:</para>"
1829  "<para>%1</para>", stateDesc);
1830 // msg == "<p>Task state:</p>"
1831 // "<p>On <i>indefinite</i> hold.</p>"
1832 ~~~
1833 
1834 If the argument and the text have different visual formats implied by

1838 
1839 ### Predefined Tags
1840 
1841 All KUIT tags belong to one of the two classes:
1842 - phrase tags, which describe parts of sentences or whole sentences
1843  inserted into running text;
1844 - structuring tags, which split text into paragraph-level blocks.
1845 
1846 A text without any structuring tags is considered equivalent of
1847 one paragraph or sub-paragraph sentence or phrase.
1848 If at least one structuring tag appears in the text,
1849 then the text is considered multi-paragraph,
1850 and no content may appear outside of structuring tags.
1851 For example:
1852 
1853 ~~~
1854 // Good:
1855 i18nc("@info",
1856  "You can configure the history sidebar here.");
1857 // BAD:
1858 i18nc("@info",
1859  "<title>History Sidebar</title>"
1860  "You can configure the history sidebar here.");
1861 // Good:
1862 i18nc("@info",
1863  "<title>History Sidebar</title>"
1864  "<para>You can configure the history sidebar here.</para>");
1865 ~~~

2136 ### Predefined Entities
2137 
2138 KUIT defines a fixed set of XML entities, which means that unlike tags,
2139 entities cannot be added to, nor their character expansions changed.
2140 The standard XML entities are:
2141 <table>
2142 <tr><th>Entity</th><th>Expansion</th></tr>
2143 <tr><td><b>\&lt;</b></td><td>less-than (`<`)</td></tr>
2144 <tr><td><b>\&gt;</b></td><td>greater-than (`>`)</td></tr>
2145 <tr><td><b>\&amp;</b></td><td>ampersand (`&`)</td></tr>
2146 <tr><td><b>\&apos;</b></td><td>single quote (`'`)</td></tr>
2147 <tr><td><b>\&quot;</b></td><td>double quote (`"`)</td></tr>
2148 </table>
2149 The `&amp;apos;` and `&amp;quot;` are really needed only
2150 within attribute values (and even there they can be avoided
2151 by choosing the opposite quote sign for quoting the attribute value).
2152 
2153 Additional entities are (Ki18n release where they were introduced
2154 given in superscript):
2155 <table>
2156 <tr><th>Entity</th><th>Expansion</th></tr>
2157 <tr>
2158 <td><b>\&nbsp;</b><sup>5.0</sup></td>
2159 <td></td>non-breaking space (`&nbsp;`)
2160 </tr>
2161 </table>
2162 
2163 The reason for not allowing custom entities
2164 can be demonstrated by the following example.
2165 An application programmer may think of defining the entity
2166 `&amp;appname;`, which would be used everywhere in text in place of
2167 the actual application name. In that way, seemingly, it would be easy
2168 to play with various names or spellings without disrupting translations.
2169 The problem, however, is that in many languages the structure of
2170 the sentence depends on the grammar properties of the particular name
2171 (e.g. its grammar gender or number), and conversely, the name may
2172 need modification according to sentence structure (e.g. by grammar case).
2173 Thus, custom entities are not allowed because
2174 they are misused too easily.
2175 If something *really* needs to be inserted verbatim into text,
2176 argument substitution is always at hand.

2181 ## Localizing Non-Text Resources
2182 
2183 It sometimes happens that a non-textual application resource
2184 needs localization. The most frequent example are images
2185 that contain some text, like splash screens.
2186 Ki18n also provides a rudimentary facility for this situation,
2187 the `KLocalizedString::localizedFilePath` static method.
2188 When called with a resource file path as the argument,
2189 this method will check what are the active languages,
2190 and look if there exists a localized version of the resource at path
2191 `<original-parent-dir>/l10n/<language>/<original-basename>`.
2192 For example, if the active language is `aa`, and a candidate image
2193 for localization is installed as:
2194 
2195 ~~~
2196 $PREFIX/share/fooapp/splash.png
2197 ~~~
2198 
2199 then a call to
2200 
2201 ~~~
2202 QString splashPath = QStandardPaths::locate(
2203  QStandardPaths::GenericDataLocation, "splash.png");
2204 splashPath = KLocalizedString::localizedFilePath(splashPath);
2205 ~~~
2206 
2207 will check if there exist the file
2208 
2209 ~~~
2210 $PREFIX/share/fooapp/l10n/aa/splash.png
2211 ~~~
2212 
2213 and return that path if it does, or else the original path.
2214 
2215 Some KDE libraries will call `KLocalizedString::localizedFilePath`
2216 on their own behind the scene, for resources that may need localization
2217 but whose paths are not directly manipulated in application sources.
2218 An example here are icons handled through `KIcon` class,
2219 which are referred to in the code only by the icon name.
