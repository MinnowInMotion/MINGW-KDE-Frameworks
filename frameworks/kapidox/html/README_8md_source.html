1 # KDE Doxygen Tools
2 
3 ## Introduction
4 
5 This framework contains scripts and data for building API documentation (dox) in
6 a standard format and style.
7 
8 The Doxygen tool is used to do the actual documentation extraction and
9 formatting, but this framework provides a wrapper script to make generating the
10 documentation more convenient (including reading settings from the target
11 framework or other module) and a standard template for the generated
12 documentation.
13 
14 
15 ## Dependencies
16 
17 ### Required
18 Python 2 or 3 is required to run the scripts. Whichever version of python you
19 use needs to have the jinja2 and yaml (or pyyaml) modules.
20 
21 The following command should install them for the current user:
22 
23  pip install --user PyYAML jinja2
24 
25 Of course, you need Doxygen!
26 
27 ### Optional
28 Doxyqml and doxypypy might be needed to let doxygen document respectevely qml
29 and python sources.
30 
31 To generate the dependency diagrams, you need the Graphviz Python bindings.
32 They are currently not available from pip, but most distributions provide them.
33 You can get binaries and source archives from
34 <http://www.graphviz.org/Download.php>.
35 
36 ## Installation
37 
38 Unlike almost every other KDE module, kapidox does not use CMake. This is
39 because it is purely written in Python, and so uses distutils. While it does
40 not need to be installed to be used (see below), you can install kapidox with
41 
42  python setup.py install
43 
44 Note: For consistency, kapidox provides a CMakeLists.txt file, but this is just
45 a wrapper around the setup.py script.
46

48 ## Writing documentation
49 
50 Writing dox is beyond the scope of this documentation -- see the notes at
51 <http://community.kde.org/Policies/Library_Documentation_Policy> and the [doxygen
52 manual](http://www.stack.nl/~dimitri/doxygen/manual/).
53 However, the script expects certain things to be present in the directory it is
54 run on.
55 
56 ### README.md
57 Most importantly, there should be a `README.md` file, like this page (backward
58 compatibility also authorize `Mainpage.dox` files). The first line of this file
59 is particularly important, as it will be used as the title of the documentation.
60 
61 ### metainfo.yaml
62 A `metainfo.yaml` file is needed for the library to be generated. It should
63 contain some metainformations about the library itself, its maintainers, where
64 the sources are etc..
65 
66 A very simple example can be:
67 
68 ```yaml
69 # metainfo.yaml
70 description: Library doing X
71 maintainer: gwashington
72 public_lib: true
73 logo: libX.png
74 ```
75 
76 A comprehensive list of the available keys can be found on
77 [this page](@ref metainfo_syntax).
78 
79 By default, the source of the public library must be in `src`, if the
80 documentation refers to any dot files, these should be in `docs/dot`.
81 Images should be in `docs/pics`, and snippets of example code should be in
82 `examples`. See the doxygen documentation for help on how to refer to these
83 files from the dox comments in the source files.
84 
85 If you need to override any doxygen settings, put them in `docs/Doxyfile.local`.
86 A global settings file is defined in `src/kapidox/data/Doxyfile.global`.
87 
88 ## Generating the documentation
89 
90 The tool for generating dox is `src/kapidox_generate`. Simply point it at the
91 folder you want to generate dox for (such as a framework checkout).
92 
93 For example, if you have a checkout of KCoreAddons at
94 ~/src/frameworks/kcoreaddons, you could run
95 
96  ~/src/frameworks/kapidox/src/kapidox_generate ~/src/frameworks/kcoreaddons
97 
98 and it would create a documentation in the current directory.
99 
100 The folders are recursively walked through, so you can also run it on
101 `~/src/frameworks` or `~/src`. For a lot of libraries, the generation can last
102 15-30 minutes and be several hundreds of Mb, so be prepared!
103 
104 Pass the --help argument to see options that control the behaviour of the
105 script.
106 
107 Note that on Windows, you will need to run something like
108 
109  c:\python\python.exe c:\frameworks\kapidox\src\kapidox_generate c:\frameworks\kcoreaddons
110 
111 ## Specific to frameworks (for now)
112 
113 You can ask `kgenframeworksapidox` to generate dependency diagrams for all the
114 frameworks. To do so, you must first generate Graphviz .dot files for all
115 frameworks with the `depdiagram-prepare` tool, like this:
116 
117  mkdir dot
118  ~/src/frameworks/kapidox/src/depdiagram-prepare --all ~/src/frameworks dot
119 
120 Then call `kgenframeworksapidox` with the `--depdiagram-dot-dir` option, like
121 this:
122 
123  mkdir frameworks-apidocs
124  cd frameworks-apidocs
125  ~/src/frameworks/kapidox/src/kapidox_generate --depdiagram-dot-dir ../dot ~/src/frameworks
126 
127 More fine-grained tools are available for dependency diagrams. You can learn
128 about them in [depdiagrams](@ref depdiagrams).
