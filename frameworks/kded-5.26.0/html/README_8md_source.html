5 ## Introduction
6 
7 KDED stands for KDE Daemon which isn't very descriptive.
8 KDED runs in the background and performs a number of small tasks.
9 Some of these tasks are built in, others are started on demand.
10 
11 ### Built in tasks
12 
13 * Checking for newly installed software and updating ksycoca when new
14  software is detected. Updating of ksycoca is done by the program kbuildsycoca
15  which gets started by kded. When kded is first started it always runs
16  kbuildsycoca to ensure that ksycoca is up to date.
17 
18 * Checking for newly installed update files. Applications can install
19  \*.upd update files. These \*.upd files are used to update configuration files
20  of users, primarily when new versions of applications are installed with
21  (slightly) different configuration file formats. Updating of configuration
22  files is done by kconf\_update. kded starts kconf\_update when it detects a
23  new update file. When kded is first started it always runs kconf\_update to
24  ensure that it has not missed any update files. kconf\_update keeps track
25  of which update files have been processed already in the config-file
26  kconf\_updaterc. It only performs a certain update once.
27 
28 ### Configuration of built in tasks.
29 
30 The built in tasks have some configuration options that can be changed by
31 editing the kdedrc configuration file. Changes need to be made with a text-
32 editor, there is no GUI available. All options are listed under the [General]
33 group:
34 
35 **HostnamePollInterval**: This can be used to adjust the time interval at which
36 the hostname is checked for changes. The time is specified in milliseconds
37 and has a default of 5000 (5 seconds).
38 
39 **CheckSycoca**: This option can be used to disable checking for new software.
40 ksycoca will still be built when kded starts up and when applications
41 explicitly request a rebuild of the ksycoca database. The user can
42 also manually rebuild ksycoca by running the kbuildsycoca program.
43 The default value of this option is "true". Checking can be disabled by
44 setting this option to "false".
45 
46 **CheckUpdates**: This option can be used to disable checking for update files.
47 kconf\_update will still be run when kded starts up.
48 The default value of this option is "true". Checking can be disabled by
49 setting this option to "false".
50 
51 **CheckHostname**: This option can be used to disable checking for hostname
52 changes. The default value of this option is "true". Checking can be
53 disabled by setting this option to "false".
54 
55 Example kdedrc file with default values:
56 
57  [General]
58  HostnamePollInterval=5000
59  CheckSycoca=true
60  CheckUpdates=true
61  CheckHostname=true
62  CheckFileStamps=true
63  DelayedCheck=false
64 
65 If FAM or DNOTIFY is not available, the filesystem will be polled at regular interval for any changes. Under the [DirWatch] group in the kdeglobals file
66 the following options are available to adjust the polling frequency:
67 
68 **PollInterval**: This can be used to adjust the time interval at which the local
69 filesystem is checked for new software or update files. The time is specified
70 in milliseconds and has a default of 500 (0.5 seconds).
71 
72 **NFSPollInterval**: This can be used to adjust the time interval at which remote
73 filesystems, such as NFS or Samba, are ebing checked for new software or
74 update files. The time is specified in milliseconds and has a default of 5000
75 (5 seconds).
76 
77 The above options are not used when FAM is used to watch for changes in the
78 filesystem, or when DNOTIFY is used. Specifying larger intervals may reduce
79 the CPU load and/or network traffic. Shorter intervals are not recommended.
80 
81 Please note that in previous versions of KDE these options where listed in
82 the kderc file.
83 
84 Example kdeglobals fragment:
85 
86  [DirWatch]
87  PollInterval=500
88  NFSPollInterval=5000

90 ### KDED modules
91 
92 Some things can be greatly simplified if they can be coordinated from a
93 central place. KDED has support for modules that will be demand-loaded
94 whenever an application attempts to make DBus call to the module.
95 This can be useful for central administration tasks.
