3 Helper library to speed up start of applications on KDE workspaces

5 ## Introduction
6 
7 kdeinit is a process launcher somewhat similar to the famous init used for
8 booting UNIX.
9 
10 It launches processes by forking and then loading a dynamic library which
11 should contain a 'kdemain(...)' function.
12 
13 Using kdeinit to launch KDE applications makes starting a typical KDE
14 applications 2.5 times faster (100ms instead of 250ms on a P-III 500) It
15 reduces memory consumption by approx. 350Kb per application.
16 
17 
18 ## How it works
19 
20 kdeinit is linked against all libraries a standard KDE application needs. With
21 this technique starting an application becomes much faster because now only the
22 application itself needs to be linked whereas otherwise both the application as
23 well as all the libraries it uses need to be linked.
24 
25 
26 ## Startup Speed
27 
28 Starting an application linked against libqt, libkdecore and libkdeui in the
29 conventional way takes approx. 150ms on a Pentium III - 500Mhz. Starting the
30 same application via kdeinit takes less than 10ms.
31 
32 (application without KApplication constructor, the KApplication constructor
33 requires an extra 100ms in both cases)
34 
35 
36 ## Memory Usage
37 
38 An application linked against libqt, libkdecore and libkdeui started in the
39 conventional way requires about 498Kb memory. (average of 10 instances) If the
40 same application is started via kdeinit it requires about 142Kb. A difference
41 of 356Kb (application without KApplication constructor)
42 
43 If we take the KApplication constructor into account, an application started in
44 the conventional way takes about 679Kb memory while the same application
45 started via kdeinit requires about 380Kb. Here the difference is somewhat less,
46 299Kb. This seems to be caused by the fact that the dynamic linker does "lazy
47 linking". We can force the linker to link everything at startup by specifying
48 "LD\_BIND\_NOW=true". When kdeinit is started with this option on, kdeinit is
49 back to its full efficiency, an application with a KApplication constructor now
50 uses 338Kb of memory. A difference of 341Kb with the normal case.

53 ## Adapting programs to use kdeinit
54 
55 The source code of a program does not require any change to take advantage of
56 kdeinit, only the build system needs to be adjusted:
57 
58 First you need to find the KF5Init package:
59 
60  find_package(KF5Init 5.0.0 REQUIRED)
61 
62 Then, instead of declaring your executable and the libraries it links to like this:
63 
64  add_executable(myexe ${myexe_SRCS})
65 
66  target_link_libraries(myexe ...)
67 
68 You must use:
69 
70  kf5_add_kdeinit_executable(myexe ${myexe_SRCS})
71 
72  # Note the different target name
73  target_link_libraries(kdeinit_myexe ...)
74 
75 
76 ## Disadvantages
77 
78 The process name of applications started via kdeinit is "kdeinit". This problem
79 can be corrected to a degree by changing the application name as shown by 'ps'.
80 However, applications like `killall` will only see "kdeinit" as process name.
81 To workaround this, use `kdekillall`, from [kde-dev-scripts][], for applications
82 started via kdeinit.
83 
84 [kde-dev-scripts]: https://projects.kde.org/projects/kde/kdesdk/kde-dev-scripts/
