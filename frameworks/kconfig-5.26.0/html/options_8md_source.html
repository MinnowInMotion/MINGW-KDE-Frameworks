1 KConfig Entry Options {#options}
2 =====================
3 
4 KConfig provides various options that can alter how it interprets configuration
5 entries on a per-entry, per-group or per-file basis. Note that these are not
6 supported by other configuration frameworks, and so should not be used in files
7 that are intended to be used by applications that do not use KConfig (such as
8 application desktop files).
9 
10 
11 Immutable Entries
12 -----------------
13 
14 KConfig's cadcading configuration mechanism usually causes values from earlier,
15 "global" configuration files to be overridden by later, "local" ones.
16 Typically, the system administrator might set global defaults, and a user might
17 override them in their local configuration files.
18 
19 However, KConfig provides a way to lock down configuration values, so that the
20 global settings override the local ones. This allows system administrators to
21 restrict the values a user can set for an entry, group of entries or an entire
22 configuration file.
23 
24 This is important for Kiosk authorization (see the KAuthorized namespace),
25 which allows parts of the user interface to be locked down.
26 
27 Configuration entries can be marked as immutable with the `$i` option. This can
28 be done on a per-entry basis:
29 
30  [MyGroup]
31  someKey[$i]=42
32 
33 on a per-group basis (which will prevent any attempts to modify entries in the
34 group at all in later files):
35 
36  [MyGroup][$i]
37  someKey=42
38 
39 or for an entire file by putting `[$i]` at the start of the file:
40 
41  [$i]
42  [MyGroup]
43  someKey=42
44  [MyOtherGroup]
45  someOtherKey=11
46 
47 Once this is done, the immutable entries or groups cannot be overridden by later
48 files of the same name (and, if the file is immutable, later files will be
49 ignored entirely).
50 
51 Note that a similar effect to file immutability can be acheived by using file
52 system permissions to prevent the user from writing to their local versions of
53 the configuration file, although (since this is normally a setup error), the
54 user will be warned that the configuration file is not writable. This warning
55 can be supressed by adding the following setting to either the relevant
56 configuration file or the `kdeglobals` file:
57 
58  [General]
59  warn_unwritable_config=true
60 
61 However, using file system permissions like this can potentially be circumvented
62 by the user if they have write access to the containing directory or can modify
63 environment variables (and `XDG_CONFIG_HOME` in particular).
64 
65 
66 
67 Shell Expansion
68 ---------------
69 
70 If an entry is marked with `$e`, environment variables and shell commands will
71 be expanded.
72 
73  Name[$e]=$USER
74  Host[$e]=$(hostname)
75 
76 When the "Name" entry is read `$USER` will be replaced with the value of the
77 `$USER` environment variable, and `$(hostname)` will be replaced with the output
78 of the `hostname` command.
79 
80 Note that the application will replace `$USER` and `$(hostname)` with their
81 respective expanded values after saving. To prevent this combine the `$e` option
82 with `$i` (immmutable) option. For example:
83 
84  Name[$ei]=$USER
85 
86 This will make that the "Name" entry will always return the value of the `$USER`
87 environment variable. The user will not be able to change this entry.
88 
89 The following syntax for environment variables is also supported:
90 
91  Name[$ei]=${USER}
92 
93 There are three environment variables that have a fallback strategy if the
94 environment variable is not set. They instead map to a location from QStanardPaths.
95 They are:
96 
97  `$QT_CACHE_HOME` - QStandardPaths::GenericConfigLocation
98  `$QT_CONFIG_HOME` - QStandardPaths::GenericConfigLocation
99  `$QT_DATA_HOME` - QStandardPaths::GenericDataLocation
